# MYPKHelper 设计方案

## 项目背景

MYPKHelper 是一个梦幻西游端游 PK 辅助工具的 Web 程序，辅助玩家在 PK 中实时计算对手的愤怒和血量。项目当前处于设计阶段，暂无代码实现。

## 技术选型

采用 **浏览器 + 本地 Python 后端** 架构。

### 选型理由

- **OCR 是核心难点**：PaddleOCR 是目前开源中文 OCR 效果最好的方案之一，Python 生态集成最简单。
- **截图直接**：`mss` 库可以精确截取指定屏幕区域，性能足够。
- **实时性好**：WebSocket 直接把识别结果推送到前端表格，延迟低。
- **打包简单**：用户最终拿到的是一个 `.exe`，双击运行自动启动后端服务并打开浏览器界面。

## 前端页面布局

界面分为三个区域：

1. **顶部监控设置区**（100% 宽度）
   - 最左侧：监控按钮（点击启动截图区域选择）
   - 居中：红色"画面监控中"状态提示（确定监控区域后显示）

2. **左侧主体表格**（占 4/5 空间）
   - 列头：名字、门派、最大气血、当前气血、护盾值、当前愤怒、叶障护盾值
   - 5 行数据（对应对方 5 个角色）

3. **右侧操作区**（占 1/5 空间）
   - 上半 3/4：日志展示区
   - 下半 1/4："战斗开始"和"战斗结束"按钮

## 战斗流程

```
1. 用户点击"监控按钮"
   → 浏览器扩展激活，打开全屏截图覆盖层
   → 用户在覆盖层上框选游戏区域，坐标传给后端

2. 后端确认收到监控区域
   → 前端显示"画面监控中"

3. 用户点击"战斗开始"
   → 后端截全图，PaddleOCR 识别 5 个角色的名字和门派
   → 初始化表格数据（根据门派设定最大气血、当前愤怒）
   → 进入高频轮询模式（每秒截图）

4. 每轮截图只截取固定区域（血量区、操作文字区）
   → OCR 识别血量变化 → 联动计算愤怒值
   → 识别操作文字 → 记录到对应角色日志
   → 通过 WebSocket 实时推送更新到前端

5. 用户点击"战斗结束"
   → 停止轮询，保存整场战斗数据到 SQLite
   → 支持查看历史战斗记录、导出 TXT/Excel
```

## OCR 识别策略

- **首场全图定位**：战斗开始时截全图，识别 5 个角色的名字、门派，以及各角色血条、操作文字区的精确位置。
- **后续分区域识别**：首场定位后，每轮只针对固定区域做 OCR，准确率更高且节省性能。
- **原因**：不同场次的战斗因阵法因素，角色位置可能变化，因此每场战斗开始时需重新定位。

## 截图触发策略

采用 **混合模式**：

- 点击"战斗开始"后，后端进入高频轮询（每秒 1 次）。
- 点击"战斗结束"或暂停时，停止轮询。
- 平时不轮询，节省 CPU。

## 数据模型

### CombatUnit（角色）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| name | string | 角色名字 |
| faction | string | 门派 |
| maxHp | number | 最大气血 |
| currentHp | number | 当前气血 |
| shield | number | 护盾值 |
| currentAnger | number | 当前愤怒 |
| yeZhangShield | number | 叶障护盾值（手动输入） |

### CombatLog（战斗日志）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| round | number | 回合数 |
| unitId | string | 关联角色 ID |
| actionType | enum | 动作类型：施法 / 受击 / 回复 / 特技 / 其他 |
| description | string | 描述 |
| hpChange | number | 气血变化量（负数为受伤） |
| angerChange | number | 愤怒变化量 |
| timestamp | number | 时间戳 |

### BattleRecord（战斗记录）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识 |
| startTime | number | 开始时间戳 |
| endTime | number | 结束时间戳 |
| opponentName | string | 对手信息（可选） |
| result | string | 战斗结果（可选） |
| unitIds | string[] | 参战角色 ID 列表 |

## 关键设计决策总结

| 决策 | 方案 | 理由 |
|------|------|------|
| 架构 | 浏览器 + 本地 Python 后端 | OCR/截图生态成熟 |
| 截图触发 | 混合模式 | 兼顾自动化与性能 |
| OCR 策略 | 首场全图定位 + 后续分区域识别 | 不同场次角色位置可能变化（阵法） |
| 区域选择 | 浏览器扩展全屏覆盖层 | 用户熟悉的截图选区体验 |
| 持久化 | SQLite 本地存储 | 轻量、无需额外安装 |
| 状态管理 | Zustand | 轻量、简洁、支持持久化 |
